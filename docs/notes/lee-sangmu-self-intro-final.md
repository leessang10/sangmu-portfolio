# 이상무 자기소개서 최종본

저는 "가치 있는 것을 제대로 만들 줄 아는 개발자"가 되는 것을 목표로 일해 온 3년차 개발자 이상무입니다.  
저에게 개발은 기술을 많이 아는 것에서 끝나지 않습니다. 사용자에게 실제로 도움이 되는지, 운영 현장에서 안정적으로 굴러가는지, 팀이 유지보수 가능한지, 그리고 비즈니스적으로 의미 있는지까지 함께 충족해야 비로소 제대로 만든 것이라고 생각합니다.

학생 시절에는 이론적으로 좋은 구조와 이상적인 구현 방식에 더 집중했습니다. 하지만 3년간 실무를 경험하며 기준이 분명해졌습니다. 이제는 "실제로 쓰이는가, 도입했을 때 효과가 있는가, 고객 관점에서 개선 체감이 있는가, 수익과 가치로 연결되는가"를 먼저 판단합니다. 이 관점 전환이 제가 지금까지 자동화, 표준화, 구조 개선을 반복해 온 가장 큰 원동력입니다.

프로젝트를 맡으면 먼저 요구사항을 받고 구현 전에 영향 범위를 정리합니다. 기존 DB와 테이블 변경 가능성, API/프론트/운영 파급 범위, 일정 리스크를 먼저 확인합니다. 이후 팀원, 상사, 클라이언트가 각각 이해할 수 있는 방식으로 설계 의도를 전달합니다. 구현은 브랜치를 분리해 로컬 테스트와 개발 서버 검증을 거친 뒤 상용 배포를 판단하고, 상용 이슈가 발생하면 빠르게 원인을 좁혀 수정/재배포하는 방식으로 책임 있게 마무리합니다.

제가 특히 중요하게 생각하는 것은 "팀 전체 생산성"입니다. 프론트엔드 동료들이 배포를 어려워하던 상황에서, 서버/브랜치 선택만으로 배포할 수 있는 자동화 스크립트를 만들어 배포 진입장벽을 낮췄습니다. 또한 API 경로, 응답, 에러 응답 형식을 표준화해 아직 구현되지 않은 API도 예측 가능한 수준으로 맞췄습니다. 이런 작업은 단순히 코드 스타일을 맞추는 일이 아니라 재작업과 커뮤니케이션 혼선을 줄이는 운영 기반이라고 생각합니다.

구조 개선에서도 같은 원칙을 유지했습니다. 빠르게 만든 기능을 그대로 방치하지 않고, 프로젝트 구조를 정리하고 커밋을 작은 단위로 분리하며 유지보수 가능한 형태로 다듬었습니다. 향후 확장을 고려해 헬스체크와 graceful shutdown 같은 운영 요소를 선제적으로 반영했고, 하드코딩 설정을 .env 기반으로 전환해 환경별 운영 안정성을 높였습니다.

Express.js 기반 코드에서 NestJS로 넘어가는 과정에서도 프레임워크 문법 자체보다 효과를 중심으로 판단했습니다. 특히 인증 로직은 단순 미들웨어 파싱에서 프레임워크 구조에 맞는 가드/데코레이터 방식으로 정리해 일관성과 재사용성을 확보했습니다. 기술을 도입할 때도 무작정 바꾸지 않고, 문제 적합성, 운영 부담, 팀 유지보수 가능성을 먼저 보고 필요할 때만 적용했습니다.

대표적으로 회사 표준 백엔드 프레임워크를 Express.js에서 NestJS로 전환한 경험이 있습니다. 계기는 명확했습니다. 레거시에는 복붙 누적, 미사용 기능 잔존, 하드코딩, API Key 평문 노출, 불명확한 네이밍 규칙, 비권장 구현 방식이 섞여 있었고 유지보수 비용이 계속 커지고 있었습니다. 저는 전환을 주도하고 핵심 설계·구현을 책임지며, 레거시 보일러플레이트를 의도 단위까지 분석한 뒤 개선된 NestJS 보일러플레이트로 점진 이관했습니다.

전환 방식은 "점진 이관 + 신규 프로젝트 선적용"의 혼합 전략이었습니다. 핵심 기능이 갖춰진 시점에 신규 프로젝트에 먼저 적용하고, 정상 운영이 가능한 것을 확인한 뒤 기존 서비스인 잡스까지 마이그레이션했습니다. 운영 중인 서비스를 옮기는 작업이었기 때문에 로그인과 핵심 서비스 기능을 중심으로 강하게 검증했고, 상용에서 발생한 예외는 핫픽스로 빠르게 대응했습니다. 그 결과 코드 구조와 네이밍 규칙이 정착되었고, 유사 API 구현 속도와 신규 인력 온보딩 속도도 좋아졌습니다.

저는 백엔드 개발자이지만 역할의 경계를 좁게 두지 않습니다. 관리자 대시보드 개발, Python 기반 크롤러/스케줄러, AWS 인프라 개선, 업무 자동화 도구 제작처럼 프로젝트 완주에 필요한 영역을 유연하게 수행해 왔습니다. 프론트에서도 픽셀이나 특정 훅 자체보다 사용자가 얼마나 쉽게 상호작용하고 예기치 못한 오류를 줄일 수 있는지를 우선합니다. 백엔드 역시 클러스터링이나 비동기 멀티프로세싱 자체가 목적이 아니라, 도입 후 무엇이 더 빨라지고 단순해지고 효율적으로 운영되는지가 더 중요하다고 생각합니다.

커뮤니케이션 방식도 실행 중심으로 다듬었습니다. 장황한 설명보다 사실 중심으로 "누가, 무엇을, 언제까지"가 바로 보이도록 정리하고, 의견이 충돌해도 감정보다 근거와 맥락으로 조율하려고 노력합니다. 제안할 때는 강요형 표현보다 "~가 좋을 것 같습니다"처럼 존중형 표현을 사용해 상대가 수용하기 쉬운 방식으로 합의를 이끌어 왔습니다.

최근에는 AI를 개발자 대체 위협으로 보기보다 생산성을 증폭하는 도구로 활용하고 있습니다. 과거에는 요구사항 정의서, 화면 설계서, 디자인, DB, API, 연동을 길게 순차 진행했다면, 지금은 AI와 목업 기반으로 모호한 요구사항을 빠르게 구체화하고 짧은 사이클로 검증/수정해 end-to-end 구현 속도를 높이고 있습니다. 다만 여기서도 기준은 같습니다. 빠르게 만드는 것만이 아니라, 운영 가능한 품질로 완성하는 것입니다.

앞으로도 저는 기술 자체를 과시하기보다, 가치 있는 문제를 정확히 정의하고 제대로 해결하는 개발자로 성장하겠습니다. 사용자, 운영, 확장, 예산, 비즈니스를 함께 보는 시야를 바탕으로 팀이 신뢰할 수 있는 결과를 꾸준히 만들겠습니다. 그리고 AI 시대에도 변하지 않는 기준인 "가치 있는 것을 제대로 만드는 힘"으로 성과를 증명하겠습니다.
