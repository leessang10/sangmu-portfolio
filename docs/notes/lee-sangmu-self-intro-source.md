# 이상무 자기소개서 소스 정리

작성일: 2026-02-20  
목적: 자기소개서/면접 답변 작성 시 사실 기반으로 빠르게 재구성하기 위한 원본 소스

## 1) 기본 프로필
- 경력: 3년차 개발자
- 중심 역할: 백엔드 개발
- 수행 범위: 프론트엔드, 데브옵스, 클라이언트 커뮤니케이션 포함
- 프로젝트 유형: SI 프로젝트 + 자사 프로젝트
- 특징: 빠르고 정확한 실행, 팀 생산성 향상 중심의 개선 습관

## 2) 실제 업무 방식 (End-to-End)
1. 프로젝트 담당 지정 후 사용자/클라이언트 요구사항 수집
2. 요구사항 반영 가능성 검토
  - 기존 DB 세팅/테이블 변경 영향 분석
  - API/프론트/운영 영향 범위 점검
  - 일정/리스크 검토
3. 설계 및 커뮤니케이션
  - 클라이언트 전달 방식 고민
  - 상사 보고 방식 정리
  - 팀원 협업을 위한 문서/기준 준비
4. 구현
  - 브랜치 분리 후 개발
  - 로컬 테스트
  - 개발 서버 배포 후 테스트
5. 배포 판단 및 상용 배포
  - 이상 없음 판단 시 상용 반영
6. 운영 대응
  - 예상치 못한 에러 발생 시 원인 파악 -> 수정 -> 재배포
7. 반복 개선
  - 자동화/시스템화
  - 기술부채 개선
  - 개인 업무계획 수립으로 일정 준수

## 3) 강점 축 (요약 키워드)
- 문제 해결: 요구사항을 구현 가능한 구조로 빠르게 전환
- 운영 관점: 배포/장애 대응/확장성까지 고려
- 협업 관점: 팀원이 바로 쓰는 문서/스크립트/표준 제공
- 개선 습관: 기술부채를 프론트/백엔드 구분 없이 지속 개선
- 학습 태도: 모르면 학습, 애매하면 질문, 동료가 어려우면 지원

## 4) 대표 사례 카드 (사실 기반)

### 사례 A. 배포 자동화 스크립트
- 상황: 프론트엔드 개발자에게 배포 작업 진입장벽이 높았음 (도메인 지식 필요)
- 행동: 선택지 기반 배포 스크립트 작성
- 스크립트 기능:
  - 어떤 서버에 배포할지 선택
  - 어떤 브랜치를 배포할지 선택
- 결과: 배포 난이도와 의존도가 낮아져 팀 협업이 쉬워짐

### 사례 B. API 규격 표준화
- 상황: API 경로/응답/에러 응답 양식이 들쭉날쭉해 예측이 어려움
- 행동: 경로/응답/에러 포맷 표준화
- 결과: 아직 미구현 API도 팀원이 예측 가능한 수준으로 일관성 확보

### 사례 C. 프로젝트 구조/코드 규율 개선
- 상황: 빠른 구현 위주로 누적된 구조 혼잡 및 기술부채
- 행동:
  - 정리된 프로젝트 구조 유지 노력
  - 커밋을 작게 분리
  - 과거 임시 구현 기능을 권장 규격으로 점진 개선
- 결과: 유지보수성과 협업 효율 개선

### 사례 D. 운영 대비 기능 선도입
- 상황: 당장은 필요 없어 보이지만 확장 시 필수인 운영 요소 존재
- 행동:
  - 헬스체크 도입
  - graceful shutdown 도입
- 결과: 수평 확장/로드밸런싱 등 이후 운영 확장 대비

### 사례 E. 인증/아키텍처 전환 적응
- 상황: Express 방식과 NestJS 방식 간 인증 처리 패턴 차이
- 행동: middleware 기반 userId 파싱에서 데코레이터/가드 방식으로 전환 적용
- 결과: 프레임워크 특성에 맞는 구조로 개선

### 사례 F. 환경변수 체계 도입
- 상황: 하드코딩 기반 운영/개발 설정 분리가 어려움
- 행동: .env 기반 환경 분리 적용 (외부 API 키/운영 환경 변수)
- 결과: 환경 전환과 설정 관리 편의성 향상

### 사례 G. 명칭 통일 리팩토링 (DB-API-프론트)
- 상황: DB는 article, 프론트는 paper 등 용어 불일치
- 행동: 도메인 명칭을 DB 기준으로 통일 제안 및 대형 리팩토링 수행
- 결과:
  - 혼동 감소
  - 불필요한 응답 변환 헬퍼 제거
  - 유지보수성 향상

### 사례 H. 범용 개발 확장
- AI 기반 일일 업무보고 자동화 도구 제작
- 관리자 대시보드 개발
- Python 크롤러 개발
- 스케줄러 구현
- AWS 기반 인프라 개선 작업 수행

## 5) 협업/커뮤니케이션 방식
- 클라이언트에게는 변경 영향과 기대 결과를 이해 가능한 언어로 전달
- 상사에게는 보고 포인트(리스크/일정/결정 필요 사항)를 분리해 전달
- 동료에게는 문서/표준/자동화로 반복 업무를 줄이는 방식 선호

## 6) 일에 대한 태도/동기
- 할 수 있는 일이 많아질수록 성장한다는 동기
- 회사에 실질적으로 도움되는 팀원이 되는 것에서 보람을 느낌
- 빠르고 정확한 판단을 위해 논리적으로 사고하고 사전 고민을 많이 하는 편
- 개인 비전: "가치 있는 것을 제대로 만들 줄 아는 개발자"가 되는 것

비전 설명 문장:
- "저의 장기 목표는 기술을 많이 아는 개발자를 넘어, 가치 있는 서비스를 제대로 완성할 줄 아는 개발자가 되는 것입니다."
- "빠르게 만드는 것과 제대로 만드는 것의 균형을 지키는 개발자가 되고자 합니다."

## 7) 자기소개서 문장 재료 (복붙용)
- "저는 구현 속도뿐 아니라 운영 가능한 구조를 만드는 백엔드 개발자입니다."
- "요구사항을 받으면 DB/API/운영 영향부터 분석해 리스크를 줄이는 방식으로 일합니다."
- "배포 자동화와 API 표준화로 팀 전체 생산성을 높이는 개선을 지속해 왔습니다."
- "프론트, 백엔드, 데브옵스 경계를 넘나들며 프로젝트 완주에 필요한 역할을 수행해 왔습니다."
- "애매한 지점은 빠르게 질문하고, 모르는 영역은 학습으로 메워 팀의 병목을 줄입니다."

## 8) 면접용 팩트 가드레일
- 모르는 수치/지표는 추정하지 않는다.
- "주도"와 "기여"를 구분해서 말한다.
- 결과는 정량이 없으면 정성 근거(협업 편의, 재작업 감소, 혼선 감소 등)로 표현한다.
- 과장형 표현(최고, 완벽, 압도적)은 사용하지 않는다.

## 9) 추후 보강하면 좋은 항목
- 프로젝트명/기간/팀 규모
- 사용 스택 상세 (NestJS, DB, AWS 구성 등)
- 실제 장애 사례 1~2개 타임라인
- 자동화 적용 전/후의 체감 변화 근거 (예: 배포 실수 감소 사례)
- 본인이 직접 작성한 문서/스크립트 링크

## 10) 추가 소스: 개인 프로젝트/기술 탐색 성향
- 개인 프로젝트를 꾸준히 진행하는 편
- 새로운 기술과 도구를 빠르게 탐색하고 실제 작업에 적용해보는 성향
- AI는 모델 자체 연구보다 "실무 생산성 도구" 관점에서 적극 활용

주요 관심 영역:
- AI 개발 도구: Claude Code 등 CLI 기반 도구
- 배포/플랫폼: Vercel, Cloudflare, AWS
- UI/프론트: shadcn, React, Svelte
- 백엔드: NestJS, Express, FastAPI, Flask
- 런타임/패키지: Node, Bun, Deno, pnpm, yarn
- 인프라 자동화: Terraform
- 기타: 크롤러/스크래핑, 하드웨어

실행 성향:
- 보일러플레이트를 만들고 지속 개선하는 작업을 자주 수행
- 회사 홈페이지 제작 같은 외주/부업성 프로젝트도 수행
- AI 활용으로 짧은 기간 내 동작 가능한 결과물을 만들어 URL로 빠르게 공유 가능

정체성 확장 문장:
- "저는 특정 직무 라벨보다 가치 있는 서비스와 프로젝트를 빠르게 완성하는 데 동기부여를 받는 개발자입니다."
- "백엔드 중심 역량을 기반으로, AI 도구를 활용해 프론트/배포/운영 경계를 유연하게 넘나드는 실행형 개발자입니다."

## 11) 면접 답변 시 주의할 균형점
- "이것저것 다 한다"는 표현 대신 "우선순위를 정해 핵심을 끝까지 완수한다"로 표현
- 기술 나열보다 실제 적용 맥락(왜 도입했는지, 무엇이 좋아졌는지) 중심으로 설명
- 빠른 제작 능력은 강조하되, 운영 안정성/유지보수 관점까지 함께 언급

권장 표현:
- "새 기술을 무작정 도입하기보다, 일정/리스크/팀 숙련도를 보고 도입 여부를 판단합니다."
- "빠르게 만들되, 배포 이후 유지보수 가능한 구조를 함께 설계합니다."

## 12) 추가 소스: 기술 도입 원칙과 실전 적용
- 개인 프로젝트에서 검증한 기술을 회사 프로젝트에 옮겨 적용한 경험이 있음
- 도입 기준은 "재미"보다 "운영 효율/팀 생산성/유지보수성"에 둠
- 무분별한 도입은 지양하고, 팀 동의와 필요성 검증 후 반영

적용 경험:
- Docker를 개인 프로젝트에서 익힌 뒤 회사 프로젝트에 도입
- TypeORM, Prisma 등 ORM을 업무에 반영
- 홈페이지 제작 경험에서 얻은 배포 설정/빌드 최적화/라이브러리 조사/베스트 프랙티스 지식을 회사 업무에 환류

도입 의사결정 방식:
1. 현재 문제를 해결하는가
2. 운영 부담을 줄이는가
3. 팀이 유지보수 가능한가
4. 팀 동의를 얻었는가
5. 필요 없으면 도입하지 않는가

면접용 문장 재료:
- "개인 프로젝트에서 충분히 검증한 기술만 업무에 보수적으로 도입해 왔습니다."
- "새 기술은 먼저 문제 적합성과 운영 부담을 비교하고, 팀 동의를 얻은 뒤 적용합니다."
- "빠른 구현 경험을 회사 표준 개선으로 연결해 실제 업무 방식까지 바꾸는 데 강점이 있습니다."

## 13) 추가 소스: 커뮤니케이션 방식
- 커뮤니케이션은 의견보다 사실 중심으로 정리
- 장황한 설명보다 "누가, 무엇을, 언제까지"가 바로 보이게 작성
- 불필요한 사정 설명/완충 문구를 줄이고 실행 중심 문장 사용
- 바쁜 상황에서도 빠르게 읽히는 요청 메시지 작성을 의식적으로 연습

업무 메시지 스타일:
- 비권장: "바쁘시겠지만 확인 부탁드립니다..."
- 선호: "OO님, XX 작업을 YY시까지 요청드립니다. 상세 요청사항은 아래와 같습니다."

갈등/협업 원칙:
- 갈등 시 주장과 근거는 분명히 제시
- 감정적으로 대응하지 않고 사실과 맥락으로 조율
- 사적으로 벽을 두기보다 팀과 자연스럽게 협력
- 동료와 상호 도움 관계를 유지하며 기술적 대화도 적극 참여
- 제안할 때는 강요형 표현보다 "~가 좋을 것 같습니다" 같은 존중형 표현 사용

면접용 문장 재료:
- "저는 상대가 바쁠수록 더 짧고 명확한 메시지가 필요하다고 생각해, 사실 중심 커뮤니케이션을 훈련해 왔습니다."
- "요청할 때는 실행 항목과 마감 시점을 먼저 전달해, 읽는 사람이 바로 행동할 수 있게 정리합니다."
- "의견 충돌이 있을 때도 감정이 아닌 근거 중심으로 대화해 팀의 의사결정 속도를 높이려 합니다."
- "제안할 때는 '~가 좋을 것 같습니다'처럼 상대를 존중하는 표현으로 방향을 맞추는 편입니다."

## 14) 추가 소스: "가치 있는 것을 제대로 만든다"의 정의
- 개인 목표: "가치 있는 것을 제대로 만들 줄 아는 개발자"
- 이 목표가 쉽지 않다는 점을 스스로 인지하고 있음
- 지금까지의 실행 습관(분석, 자동화, 표준화, 협업, 운영 대비)의 원동력은 이 목표

가치 있는 것의 기준:
1. 사용자/고객에게 실제로 문제를 해결해 주는가
2. 팀과 회사 입장에서 유지 가능한가
3. 비용 대비 효과가 있는가

"제대로 만든다"의 기준:
1. 요구사항을 정확히 반영한다
2. 운영 관점에서 안정적으로 굴러간다
3. 확장 관점에서 미래 변경에 견딜 수 있다
4. 예산 관점에서 과도한 비용/복잡도를 피한다
5. 고객 관점에서 사용성과 신뢰를 해치지 않는다

핵심 서술 문장:
- "저에게 개발은 기능 구현 자체가 아니라, 여러 관점에서 오래 버티는 결과물을 만드는 일입니다."

## 19) 추가 소스: 업무 방식 전환 (문서 중심 -> AI 기반 실행 중심)

5. 화면 구현
6. API 구현

3. 목업 기준으로 DB 설계와 API 설계를 빠르게 확정
4. 구현과 연동을 짧은 사이클로 반복

핵심 변화:

- 혼자서도 서비스에 신규 기능을 end-to-end로 추가 가능한 실행력 확보

면접/자소서 문장 재료:

- "설계서가 완벽해질 때까지 기다리기보다, 먼저 작동하는 프로토타입으로 방향을 검증하고 DB/API를 짧은 사이클로 확정합니다."
- "이 방식 덕분에 혼자서도 신규 기능을 end-to-end로 빠르게 추가할 수 있는 실행력을 갖추게 되었습니다."

## 20) 추가 소스: Express.js -> NestJS 회사 표준 전환 계기 (잡스)
- 역할: 전환을 주도하고 핵심 설계·구현을 책임

전환 배경(문제 인식):
- JS 기반 레거시에서 오류를 런타임 이후에 발견하는 경우가 많았음
- 여러 프로젝트를 거치며 복붙식으로 남은 미사용 기능이 누적됨
- 하드코딩, API Key 평문 노출, 불명확한 네이밍 규칙이 섞여 유지보수 난도가 높았음
- 비권장 구현 방식이 축적되어 코드 신뢰도와 일관성이 떨어졌음

전환 판단:
- 목표는 프레임워크 교체 자체가 아니라 유지보수 가능한 표준 확립
- TypeScript 기반으로 오류를 더 빠르게 캐치하고, 구조/규칙/보안을 함께 정리하는 방향으로 결정

실행 전략(혼합):
1. 레거시 보일러플레이트를 의도 단위까지 분석/정리
2. 개선 버전 NestJS 보일러플레이트로 기능을 점진 이관
3. 핵심 기능이 갖춰진 시점에 신규 프로젝트부터 우선 적용
4. 운영 안정성 확인 후 기존 레거시 프로젝트도 순차 마이그레이션

적용/확산:
- 신규 적용 후 정상 운영이 가능하다고 판단된 뒤, 기존 서비스 `잡스`까지 신규 보일러플레이트 기반으로 이관

리스크와 대응:
- 가장 큰 리스크는 운영 중인 서비스 이관(달리는 마차의 바퀴 교체)
- 개발 서버에서 충분히 테스트하고, 특히 로그인/핵심 서비스 기능을 집중 검증
- 상용에서 발생한 예기치 못한 오류는 핫픽스로 빠르게 대응

체감 효과:
1. 구조/네이밍 규칙 정착으로 코드 이해와 협업 품질 향상
2. 패턴 재사용 기반으로 신규 API 개발 속도 개선
  - 예: 회원 목록 API 패턴을 참고해 댓글 목록 API를 유사 구조로 빠르게 구현
  - 페이지네이션, join, 필터/정렬, userId 획득 로직 분리가 잘 되어 재사용성이 높았음
3. 신규 인력 온보딩 속도 개선
  - 예: 스프링 경력 동료도 구조를 빠르게 이해하고 적응
4. 하드코딩/평문 노출 등 보안/설정 관리 위험 요소 정리

3줄 요약:
1. Express 레거시의 유지보수/보안/일관성 문제를 해결하기 위해 NestJS+TS 표준 전환을 추진했다.
2. 점진 이관 + 신규 선적용 전략으로 리스크를 낮추고, 검증 후 `잡스`까지 확산했다.
3. 결과적으로 구조 표준화, 개발 속도, 온보딩, 운영 안정성이 함께 개선됐다.

60초 답변:
- "회사 표준 백엔드를 Express에서 NestJS로 바꾼 이유는 기술 유행이 아니라 유지보수 비용 문제였습니다. 레거시에는 복붙 누적, 미사용 기능, 하드코딩, 평문 키 노출, 네이밍 붕괴가 섞여 있었고 JS 특성상 오류를 늦게 잡는 경우도 많았습니다. 저는 기존 보일러플레이트를 의도 단위까지 분석한 뒤, NestJS+TS 기반 개선 보일러플레이트로 기능을 점진 이관했습니다. 핵심 기능이 갖춰졌을 때 신규 프로젝트부터 적용하고, 안정 운영을 확인한 뒤 기존 서비스 잡스까지 마이그레이션했습니다. 운영 중 이관이라 로그인/핵심 기능을 집중 검증했고 상용 이슈는 핫픽스로 대응했습니다. 결과적으로 코드 규칙이 정착되고 유사 API 재사용이 쉬워져 개발/온보딩 속도가 개선됐습니다."

STAR 카드:
- Situation: Express 레거시의 구조 붕괴와 유지보수 비용 증가
- Task: 운영 중단 없이 NestJS 표준으로 전환해 품질/속도/안정성 확보
- Action: 레거시 분석 -> 개선 보일러플레이트 설계 -> 점진 이관 + 신규 선적용 -> 운영 검증/핫픽스
- Result: 잡스 포함 서비스 전환 완료, 코드 표준화와 개발 재사용성/온보딩 개선

전환 사례 가드레일:
- 과장 금지: 확인 불가 수치 대신 관찰 가능한 정성 근거 사용
- 표현 기준: "전환을 주도하고 핵심 설계·구현을 책임"으로 고정
- 비하 금지: 기존 코드/팀 비난 없이 개선 맥락 중심으로 설명
